# Client surface checks for generated SDKs

_Verify properties of generated SDKs that the wire tests can't observe —
described once in the spec, precomputed centrally, checked per emitter. Mostly
deterministic; AI only for the fuzzy residual._

## What this is

Spector verifies emitter parity on the **wire**: it generates clients, runs them
against mock APIs, and confirms the bytes are right. A large class of behavior
never reaches the wire — whether an internal type is actually hidden, whether a
property was renamed for the client, whether a method was relocated, whether a
model was flattened. Today those are checked by hand, per language.

**Client surface checks** close that gap. You annotate a spec element with
`@clientSurfaceDoc("…")` — a language-agnostic description of what should be true
of the generated client surface. A precompute step collects every
`@clientSurfaceDoc` into a shared **checks doc**. Each emitter supplies a
**context.md** (what those concepts look like in its output) and a
**signatures.json** (mechanical shortcuts). A deterministic runner checks
everything it can; only genuinely fuzzy items go to the `/check-client-surface`
prompt.

It is not limited to client naming — `@clientSurfaceDoc` can assert any
generated-surface property (access/visibility, shape, naming, …).

## The pieces and where they live

| Piece                                                                                          | Lives in                                  | AI?                                               |
| ---------------------------------------------------------------------------------------------- | ----------------------------------------- | ------------------------------------------------- |
| `@clientSurfaceDoc` decorator                                                                  | spector (with `@scenario`/`@scenarioDoc`) | no                                                |
| checks doc (precomputed manifest of all `@clientSurfaceDoc`)                                   | generated by spector, shipped with specs  | no                                                |
| `context.md` (concept prose + Emitter facts)                                                   | each emitter repo                         | no (authored)                                     |
| `signatures.json` — **routing table** (which categories are deterministic)                     | each emitter repo                         | no                                                |
| `verify.py` — **deterministic check tool**, run once per deterministic item                    | each emitter repo / shared                | no                                                |
| `/check-client-surface` prompt — **the orchestrator** (routes each check to `verify.py` or AI) | monorepo root `.github/prompts/`          | one session; AI only for non-deterministic checks |

## The `@clientSurfaceDoc` decorator

Mirrors `@scenarioDoc`. It attaches a language-agnostic assertion to any spec
element, plus an optional `category` tag emitters map to a code signature.

```tsp
@clientSurfaceDoc(
  "The enum is generated as internal — not part of the public API surface — in the target language.",
  "access"
)
@access(Access.internal)
enum SecretCodes {
  granted,
  denied,
}
```

The text describes intent ("internal", "renamed", "flattened"); the emitter's
context.md / signatures.json translate that into "what internal looks like in
Python".

## Precompute step: the checks doc

Analogous to spector's `generate-scenarios-summary`. A deterministic walk over
all specs (`listClientSurfaceDocs(program)`) collects every `@clientSurfaceDoc`
into a manifest — one item per annotation, recording a stable id, the scenario,
the target symbol's identity, the category, and the assertion. See
`spector/generate-client-surface-checks.ts` and the example
`spector/client-surface-checks.generated.{md,json}`. Runs **once per spec change,
no AI**, centrally in spector, and ships with the specs.

## Running it

One call. In Copilot Chat (agent mode) or via the CLI:

```
/check-client-surface   →   emitter: http-client-python   flavor: azure   scenario: (blank = all)
```

The prompt is an **orchestrator**. In one agent session it walks the checks doc
and, for each check, looks at `signatures.json`: if the category is listed it
**runs `verify.py`** (a real code check that returns a JSON verdict) and uses
that result; if not, it **verifies the check itself** (AI). It prints one merged
table marking each row `deterministic (code)` or `ai`.

So the deterministic checks are genuinely decided by code, not the model — the
AI is the orchestrator (looping, routing, merging) plus the verifier for the
categories with no routine. One session = one invocation regardless of how many
checks.

### Runnable demo (three examples)

`demo-checks.json` is a self-contained manifest with the three gist examples,
checked against the real generated Python SDKs under `tests/generated/azure`:

| id | category | how |
| --- | --- | --- |
| `Client_Naming_UnionEnum_name` | `naming` | deterministic (`identifier_casing_insensitive`) |
| `Client_Location_deleteUser_on_Admin` | `client-location` | deterministic (`operation_client_membership`) |
| `Client_Hierarchy_Dog_extends_Pet` | `hierarchy` | **AI by default** (no signature) → deterministic once promoted |

Run the deterministic batch (naming + client-location pass; hierarchy lands in
`needs_ai`):

```bash
python eng/scripts/client-criteria/verify.py --batch \
  --checks eng/scripts/client-criteria/demo-checks.json \
  --signatures eng/scripts/client-criteria/signatures.json \
  --generated-root tests/generated --flavor azure --language python
```

Then show the objective/subjective boundary is set by `context.md`, not the
scenario — rerun with `signatures.promoted.json`, which adds the one `hierarchy`
entry, and the same hierarchy check flips from AI to a deterministic PASS with no
change to the spec assertion:

```bash
python eng/scripts/client-criteria/verify.py --batch \
  --checks eng/scripts/client-criteria/demo-checks.json \
  --signatures eng/scripts/client-criteria/signatures.promoted.json \
  --generated-root tests/generated --flavor azure --language python
```

## Per-emitter files

The one required per-emitter file is **context.md** — Emitter facts (language,
generated-root, flavors, checks-doc path) plus **concept signatures**: what each
concept looks like in this language's code, written concretely enough that the
prompt can execute it as a file check rather than guess.

Optionally, **signatures.json** (machine-readable mirror of the mechanical
signatures) plus the shared **verify.py** give a fully non-AI runner for CI.

## Optimize for precompute, minimize AI calls

AI is the fallback, not the default. Work is pushed left through four layers:

1. **Manifest precompute (no AI, central, once per spec change)** — the
   `@clientSurfaceDoc` walk is deterministic. The list of _what to check_ is free.
2. **Concept signatures (no AI, authored once per language)** — ~a dozen entries,
   reused across all scenarios, not per-scenario authoring.
3. **Deterministic checks at verify time (no AI)** — `verify.py` runs every item
   whose category has a `signatures.json` entry by reading files. This covers the
   bulk.
4. **AI only for the fuzzy residual, cached and scoped** — `/check-client-surface`
   handles only items with no mechanical signature. Cache each result on
   `(item id + hash of the relevant generated file + model version)`; scope a run
   to packages changed in the diff.

In the runnable demo above, 2 of 3 items resolve deterministically and the
`hierarchy` item routes to AI — until a language authors its `hierarchy`
signature (see `signatures.promoted.json`), after which all 3 are deterministic.

## Onboarding a language

One required file: `context.md` under
`packages/<emitter>/eng/client-surface-checks/` — its Emitter facts plus concept
signatures. The decorator, the checks doc, and the `/check-client-surface` prompt
are shared. A new emitter writes no assertions and no per-scenario criteria — it
only describes what its own concepts look like in code. Optionally add
`signatures.json` if you also want the zero-AI `verify.py` path in CI.

## Open questions

- Where does the precomputed checks doc ship — a consumable asset in spector, or
  alongside the specs in http-specs?
- Should the manifest record the **resolved per-language symbol name** (so even
  renamed-and-access-controlled types stay mechanically checkable)?
- Should `signatures.json` carry executable patterns (glob/regex) so a fully
  non-AI runner can execute them, with the prompt purely as the fuzzy fallback?
- How do items mark **N/A** for a language whose surface has no such concept?
